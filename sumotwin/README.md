# 7065twinv2 — SIG#7065 twin, canvas for the NE channelized free-right

> **Duplicate of `7065twin/`.** The network is currently **identical to base**
> (regenerated by the same `digitize.py` → `netconvert` flow,
> `bash 7065twinv2/build.sh`). v2 exists to add **one thing**: the real
> channelized free-right slip lane that base intentionally omitted.
>
> ## The slip lane to add — trace it in netedit
> **Movement:** Peachtree eastbound → **right turn** → Piedmont **northbound**,
> a **free-right that bypasses the signal** (does not pass through J). In twin
> edge terms that is **`NE_in` → `N_out`** (a right turn — it comes off the NE
> Peachtree approach and drops onto the Piedmont north leg).
>
> A first attempt to place this from bearing math put it on top of a building
> (`aerial_overlay.png`), so it was removed — a slip's real curb can only be
> **traced from the aerial**, not computed. To do that:
> ```bash
> cd 7065twinv2
> netedit 7065.net.xml -g 7065_aerial.view.xml   # georeferenced aerial under the net
> ```
> 1. Zoom to the corner between the NE approach and the N leg.
> 2. **Create-edge mode** (`E`): click along the real slip curb — start on/next to
>    `NE_in`, trace the curve, end on `N_out`. Clicking an existing lane splits it
>    and makes a junction. This new edge is the slip; it never touches `J`, so it
>    is inherently a signal bypass.
> 3. **Inspect** (`I`): set the slip edge to `numLanes=1`, speed, allowed classes.
> 4. **Connection mode** (`C`): wire the `NE_in` right lane → slip → `N_out` lane.
>    Leave the slip's end junction `priority` so the merge yields (free right).
> 5. `Ctrl+S` — saves straight into `7065.net.xml`.
>
> For a **reproducible** build instead, trace it in netedit then send the slip
> edge's `shape` coords back to bake into `digitize.py`.
>
> ### Aerial-tracing files (v2-only)
> | File | What |
> |---|---|
> | `7065_aerial.view.xml` | Puts `satellite_ref.png` under the net, aligned (decal computed from the tile bounds + net `netOffset`). |
> | `aerial_overlay.png` | Verified fit of net-over-aerial; also shows where the removed bearing-math slip wrongly landed. |

# (inherited from 7065twin) — SUMO digital twin of GDOT SIG#7065

Signalized intersection **Peachtree Road NE × Piedmont Road NE** (SR 141 @ SR 237),
Buckhead, Atlanta GA. Center **33.84373150160545, -84.37140379788055**.
Approach length (from `J`, the junction center node — not the stop bar) is
**100 m by default**, per-leg overrides in `LEG_LEN`: top (N) 100 m, bottom (S)
70 m, left (SW) 70 m, right (NE) 105 m.

## Why this is re-digitized from the aerial (not OSM)
The first attempt imported OpenStreetMap. It turned out OSM **under-maps this
intersection**: the **Piedmont north leg** (`way 525767115`) has *no* `lanes` or
`oneway` tags, so netconvert rendered it as a 1-lane alley, and its geometry is
simplified/misaligned versus reality. Peachtree and the Piedmont *south* leg were
fine, but the north leg made the twin wrong.

So this twin is **digitized directly from a georeferenced satellite tile**
(`satellite_ref.png`, Esri World Imagery, known bounds): the junction center and
one centerline point per leg were picked off the aerial, converted pixels → lon/lat
→ local meters, and rebuilt as clean straight 100 m approaches on the measured
bearings — the same OSM→NETEDIT correction workflow the APSIPA paper used, done
programmatically. Evidence images are kept in this folder (see below).

## Ground-truth lane configuration (inbound → J / outbound ← J)
| Leg | Road | Inbound | Outbound |
|---|---|---|---|
| N  | Piedmont Rd NE | **4** | 2 |
| S  | Piedmont Rd NE | **4** | 2 |
| NE | Peachtree Rd NE | **5** | 3 |
| SW | Peachtree Rd NE | **5** | 3 |

S/SW/NE lane-use is netconvert's default (1 right + n straight + 1 left) except
NE (2 left, 3 straight, no right — see `digitize.py`) and SW (2 left, 2
straight, 1 shared straight/right). (Lane counts from Google Maps + the
correctly-tagged OSM south & Peachtree legs.)

## N-leg (top): late left-turn pocket
Rather than a left-turn lane running the full 100 m, the N approach is split
into 2 segments: `N_far` (`nN`→`nNleft`, 100→40 m, 3 lanes) and `N_near`
(`nNleft`→`J`, 40→0 m, 4 lanes) — the dedicated left-turn lane only exists on
`N_near` (last **40 m**, see `N_LEFT_DIST` in `digitize.py`); for the first
60 m there are just 3 lanes.

## Open it
```bash
sumo-gui -c 7065twin/7065.sumocfg      # View → "real world" scheme
sumo     -c 7065twin/7065.sumocfg      # headless (loads clean, no demand yet)
netedit  7065twin/7065.net.xml         # inspect/adjust lanes, connections, TLS
```

## Rebuild
```bash
bash 7065twin/build.sh                  # digitize.py -> netconvert ; no network needed
```

## Files
| File | What it is |
|---|---|
| `7065.net.xml` | **Final network** — 1 signal, 100 m legs, real lane counts |
| `7065.sumocfg` / `7065.view.xml` | Runnable config + sumo-gui view |
| `digitize.py` | Turns the aerial picks into `plain/7065.nod.xml` + `.edg.xml` |
| `plain/` | Digitized node/edge source fed to netconvert |
| `preview.png` | SUMO-style render of the final twin |
| `redigitized_overlay.png` | **Final net overlaid on the satellite** (fidelity check) |
| `satellite_ref.png` | Georeferenced aerial the geometry was picked from |
| `fidelity_osm_vs_sat.png`, `northleg_check.png` | Evidence of the OSM data gap |
| `osm_raw.osm.xml` | The rejected OSM download (kept for reference) |

## To adjust geometry
Edit the pixel picks / lane counts at the top of `digitize.py` (`J_PX`, `LEG_PX`,
`LANES`) and re-run `build.sh`; re-check against `redigitized_overlay.png`. For
finer curvature, open `7065.net.xml` in NETEDIT.
